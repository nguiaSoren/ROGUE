# Frontend architecture

Engineering map of the ROGUE Next.js dashboard (App Router, `frontend/src/app/`).
Live at `https://rogue-eosin.vercel.app`; consumes the FastAPI backend at
`https://rogue-private.onrender.com` (`NEXT_PUBLIC_API_BASE`).

> This is Next.js with breaking changes from older versions — read
> `node_modules/next/dist/docs/` before writing code (see `AGENTS.md`).

## Backend access — `src/lib/api.ts`

All public-dashboard reads go through `api.*` in `src/lib/api.ts`, which wraps the backend
endpoints exposed by `src/rogue/api/main.py`. Centralized concerns:

- **ISR caching:** `apiGet` fetches with `next: { revalidate: 300 }` — pages render from a
  5-minute cache instead of paying the Vercel→Render→Neon round-trip per view. New Neon data
  surfaces within the revalidate window.
- **Cold-boot resilience:** Render's free tier returns transient 502/503/504 (or drops the
  socket) on cold start; `apiGet` retries gateway statuses + network throws (2 retries,
  1.5s/3s backoff) with a 12s per-attempt timeout so a cold start never paints "unavailable".

`api` methods: `health`, `attacks`, `attackDetail`, `breachMatrix`, `breachCell`, `brief`,
`banditStats`, `personaStats`, `escalationStats`, `mutationStats`, `stubbornnessStats`.

## Live routes → components → endpoints

| Route | Key components | Endpoints consumed |
|---|---|---|
| `/` (`page.tsx`) | landing / dashboard overview | `api.health`, `api.attacks`, `api.breachMatrix`, `api.banditStats`, `api.personaStats`, `api.escalationStats`, `api.mutationStats`, `api.stubbornnessStats` |
| `/feed` (`feed/page.tsx`) | discovery feed | same broad set as `/` + `api.brief` |
| `/matrix` (`matrix/page.tsx`) | breach-matrix grid | `api.breachMatrix`, `api.stubbornnessStats` |
| `/matrix/cell` (`matrix/cell/page.tsx`) | `CellView` (client) | `GET /api/breaches/cell` (fetched client-side in `CellView`, not server-side — a dynamic route, so SSR is skipped to avoid a hard 502 on a Render cold-cycle; `CellView` does retry/timeout/"waking up" itself) |
| `/brief` (`brief/page.tsx`) | threat-brief view | `api.brief` |
| `/analytics` (`analytics/page.tsx`) | telemetry report | bundled static `/analytics.json` (NOT the live API — regenerated by `scripts/ops/build_analytics.py`, published via `scripts/ops/publish_analytics.sh`; client `fetch("/analytics.json")`) |
| `/product` (`product/page.tsx`) | marketing product tour (`marketing/preview/*` components) | none (static marketing previews) |

**Live SSE widget.** The root layout (`src/app/layout.tsx`) wraps the app in
`SseFeedProvider` (`src/components/sse-feed-provider.tsx`), a single shared `EventSource` to
`${API_BASE}/api/sse/feed?since_days=2`. The backend sends one `snapshot` event per
connection then heartbeats; new snapshots arrive on reconnect. Consumers
(`nav.tsx`, `live-attack-ticker.tsx`, the count pill) read it via `useSseFeed()` — one
connection instead of three. `Nav` also polls `api.health` to show DB up/down.

## Commercial pages + the `COMMERCIAL` flag

`src/lib/flags.ts` exports `COMMERCIAL = process.env.NEXT_PUBLIC_SHOW_COMMERCIAL === "true"`
— a build-time, global flag (baked into the build; flipping it requires a Vercel redeploy).

- **Default (honest hiring mode):** `/pricing` 404s (`notFound()`), and the
  pricing/enterprise/security nav links are hidden (`nav.tsx` gates them on `COMMERCIAL`).
  Commercial CTAs route to `/early-access` instead.
- **`COMMERCIAL=true` (startup pitch):** `/pricing` is reachable and the
  pricing/enterprise/security nav links + CTAs appear.

Commercial/marketing routes: `/pricing` (flag-gated), `/enterprise`, `/security`, `/about`,
`/product`, `/resources`, `/case-studies` + `/case-studies/[slug]`, `/deck`, `/early-access`,
`/request-demo`. Lead-capture pages (`/request-demo`, `/early-access`, newsletter) post to
backend tables `demo_requests` / `newsletter_subscribers` (see `docs/db_schema.md`).

## Scan flow (authenticated product shell)

The `(app)` route group (`src/app/(app)/layout.tsx`) is the authenticated shell. "Session" =
the customer's API key in an httpOnly cookie (`src/lib/session.ts`); a missing key
`redirect("/sign-in")`s. Sign-in is at `/sign-in`; the session cookie is set via
`POST /api/session`.

Routes: `/scans` (list), `/scans/new` (launch), `/scans/[scanId]` (live progress),
`/scans/[scanId]/report` (report).

Internal Next API routes under `src/app/api/scans/` are **server-side proxies** that attach
the session API key (kept off the browser) and forward to the platform `/v1/*` backend:

| Next route | Forwards to |
|---|---|
| `POST /api/scans` | `POST /v1/scans` |
| `POST /api/scans/validate` | validate a ScanSpec before launch |
| `GET /api/scans/[scanId]` | `GET /v1/scans/{id}` (live progress polling) |
| `GET /api/scans/[scanId]/report` | the scan report |
| `POST /api/scans/[scanId]/cancel` | cancel a running scan |
| `POST /api/session` | set the session cookie from a submitted API key |
| `POST /api/revalidate` | on-demand ISR revalidation |

`/scans/new` validates against `/api/scans/validate`, then submits to `/api/scans`.
